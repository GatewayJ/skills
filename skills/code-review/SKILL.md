# Code Review

## 概览

执行偏资深工程视角的代码审查，优先发现真实缺陷、架构漂移、缺失测试、正确性漏洞、性能风险和一致性问题。代码风格问题只在影响可维护性、可读性、API 边界或后续演进成本时作为审查意见输出。

## 审查流程

1. 以调用提示词指定的范围为准：可能是当前工作区修改、指定 commit、PR、分支、diff、补丁或文件列表。不要在 skill 内固定范围推断规则；如果提示词没有给出可审查范围，先用一个简短问题向用户确认。
2. 阅读足够的上下文，理解项目已有架构、模块职责、依赖方向、命名习惯、日志风格和测试风格。
3. 默认读取 [references/review-checklist.md](references/review-checklist.md) 作为审查清单。只有纯文档、格式化或极小范围变更可以跳过，并在输出中说明。
4. 先建立“当前变更相关的架构形态”，再判断本次变更是否贴合。涉及架构时，必须说明相关模块的业务语义、架构位置、依赖方向和形态变化。
5. 尽可能运行或定位相关测试、静态检查、构建命令。无法运行时，明确说明原因和残余风险。
6. 输出时先列问题，按严重程度排序。每条问题都要包含文件行号、当前行为、风险影响和可执行的修复方向。
7. 区分已确认缺陷和开放问题。没有本地证据支撑的猜测不要写成确定问题。

## 审查维度

### 架构边界

检查变更是否贴合项目已有架构形态：

- 识别当前层次、模块、适配器、领域边界和依赖方向，并说明这些模块承担的业务语义。
- 说明被改模块位于什么架构位置：入口层、编排层、领域层、应用服务层、仓储层、适配器层、基础设施层、后台任务、外部集成或共享库。
- 找出是否新增了架构层、服务边界、全局依赖、跨层调用或共享抽象。
- 判断新增形态是否由真实复杂度驱动，还是引入了偶发性结构。
- 参考设计模式、知名架构风格和架构原则，但不要机械套用。重点看 SRP、OCP、ISP、DIP、LoD、CRP 等原则是否帮助解释真实风险。
- 涉及架构变化时，给出形象化文字描述：

```text
业务语义: 订单取消流程需要校验订单状态、释放库存、记录审计事件
架构位置: API handler -> application service -> domain policy -> repository -> storage
变更前: handler 只做入参和鉴权，取消规则集中在 application service/domain policy
变更后: handler 新增库存释放和状态判断，绕过 domain policy
风险: 入口层开始拥有领域决策，形成“绕过边界”，后续 CLI/worker 入口会出现重复规则
```

### 代码风格和 API 边界

审查可维护性，而不是个人偏好：

- 函数是否过长、分支过多，或承担了多个不相干职责。
- 函数或方法参数是否过多，尤其是多处一起传递的一组参数是否应该建模为值对象或 options。
- 对外暴露是否克制：导出符号、public 方法、配置字段、API 和包级 helper 只有在边界外确实需要时才应该存在。
- 逻辑是否比问题本身更复杂，或简单到遗漏必要的校验、状态处理和失败处理。
- 是否存在冗余条件、重复转换、重复错误处理、复制粘贴的业务规则或重复代码。
- 是否违反 SRP、OCP、ISP、DIP、LoD、CRP 等原则，并且这种违反会带来实际维护或演进风险。
- 优先采用项目已有局部习惯，不强行套用通用风格规则。

### 注释、日志和可观测性

- 检查注释是否过时、与代码矛盾、描述已经删除的行为，或只是在复述显而易见的实现。
- 保留解释非显然约束、不变量、取舍和协议行为的注释。
- 关键路径应打印足够诊断上下文：请求标识、资源标识、状态迁移、外部调用失败、重试耗尽、权限失败和一致性失败。
- 标记泄露敏感信息、刷屏热路径、吞掉错误或丢失因果上下文的日志。

### 单测

- 检查变更行为和失败模式是否有测试覆盖。
- 判断测试是否真正约束预期行为，而不是只覆盖 happy path 或实现细节。
- 警惕“为了过单测而修改业务逻辑”：不能接受为了测试方便而削弱生产行为的变更。
- 优先要求能在回归时失败的测试。
- 能运行相关测试时优先运行；不能运行时说明是依赖缺失、环境缺失、范围过大、耗时过长还是用户未授权。

### 功能正确性、性能和一致性

- 对照需求和周边契约验证实现行为。
- 检查边界情况：nil、空值、错误传播、重试、部分失败、取消、超时、并发、顺序、幂等、升级和迁移路径。
- 检查性能风险：循环中的重复 I/O、N+1 查询、无界内存增长、大输入低效处理、锁竞争和过度序列化。
- 检查一致性风险：缓存陈旧、事务边界、最终一致性假设、非原子更新、竞态窗口和分布式状态漂移。

## 输出格式

默认使用用户的语言；中文用户用中文回复。

先输出问题。架构形态、测试、开放问题按需输出：没有架构相关变化就省略架构形态；没有测试动作或测试缺口就省略测试；没有开放问题就省略开放问题。

```markdown
**问题**
- High: [file:line] 当前变更会 ...，因为 ...。建议 ...
- Medium: [file:line] ...

**架构形态**
业务语义: ...
架构位置: ...
变更前形态: ...
变更后形态: ...
判断: ...

**测试**
已运行: ...
缺口/残余风险: ...

**开放问题**
...
```

如果没有发现问题，直接说明未发现明确问题，并补充仍然存在的测试缺口或残余风险。

## 严重级别

- Critical：可能导致数据丢失、安全漏洞、权限绕过、大面积生产故障或不可逆状态破坏。
- High：核心路径功能错误、跨租户/跨用户影响、事务或一致性破坏、明显性能退化，或架构边界被破坏到会快速扩散。
- Medium：边界场景错误、测试缺口、可维护性风险、局部性能问题、局部架构漂移或未来扩展成本明显上升。
- Low：局部可读性、命名、注释、日志质量或小范围重复代码问题；只有在影响维护判断时输出。
