# 代码审查清单

用于范围较大、风险较高或上下文不清的代码审查。不要机械输出每一个清单项；只报告会造成真实风险或明显维护成本的问题。

## 架构边界审查

- 项目当前的架构形态是什么？
- 当前已经存在什么层次：UI/API、命令处理、领域服务、应用服务、仓储、适配器、基础设施、worker、scheduler、存储、外部 client？
- 被修改模块分别承担什么业务语义？例如鉴权、编排、领域决策、状态迁移、持久化、缓存、消息投递、外部系统适配或展示。
- 被修改模块位于什么架构位置？它应该只做什么，不应该做什么？
- 本次变更是否新增了层、包、服务、抽象、全局注册点、框架依赖或依赖反转点？
- 新层次的命名来自稳定领域概念，还是来自临时实现细节？
- 依赖方向是否仍符合项目局部约定？
- 领域逻辑是否泄露到 handler、controller、CLI、数据库适配器或 UI 代码里？
- 基础设施代码是否反向调用领域或应用代码，导致循环依赖或隐式耦合？
- 编排逻辑是否由一个清晰位置负责，还是散落在 handler、service 和 repository 之间？
- 本次变更是否用不同名字重复了已有架构角色？
- 当前架构还能否简单描述，还是需要额外例外规则才能讲清楚？

## 架构形态描述

架构相关问题必须给出简短形象化描述。优先使用文字图，而不是长段解释：

```text
业务语义:
模型发布流程需要校验权限、解析版本、写入元数据、触发异步构建。

架构位置:
API handler 是入口层，publish service 是应用编排层，repository 是持久化边界，builder client 是外部适配器。

当前形态:
API/CLI -> application service -> domain model -> repository interface -> storage adapter

变更后形态:
API/CLI -> workflow manager -> application service -> repository interface -> storage adapter

判断:
新增 workflow manager 同时拥有重试和持久化状态职责，和 application service 职责重叠。
```

常用形态标签：

- 流水线：每一阶段转换数据并传给下一阶段。
- 分层蛋糕：上层只依赖下层，依赖方向稳定。
- 中心辐射：一个协调者分发到多个独立适配器。
- 六边形：领域核心被端口和适配器隔离。
- 网状调用：模块互相随意调用；除非项目明确采用，否则通常是坏味道。
- 双脑：两个地方同时拥有同一个决策或状态迁移。
- 绕过边界：调用方跳过既定边界，直接触达更底层模块。

## 架构原则和模式参考

这些原则用于帮助解释风险，不是为了机械套标签。只有当原则违反会带来实际缺陷、扩散成本或演进阻力时，才作为审查问题输出。

- SRP，单一职责原则：一个模块、函数或类是否只有一个清晰变化原因？如果 handler 同时负责鉴权、业务决策、持久化和外部调用，通常说明职责聚合过多。
- OCP，开闭原则：新增业务类型、策略或后端时，是否主要通过扩展完成，而不是反复修改核心分支？但不要为了未来可能性过早抽象。
- ISP，接口隔离原则：调用方是否被迫依赖自己不用的方法、字段或配置？大而全接口、胖 client、通用 manager 往往会拖大变更面。
- DIP，依赖倒置原则：高层业务策略是否依赖低层实现细节？领域或应用层直接依赖数据库、HTTP 框架、云 SDK 时要检查边界是否被穿透。
- LoD，迪米特法则：调用方是否穿透多层对象结构拿内部细节？长链式访问、跨对象改内部状态通常说明封装不足。
- CRP，共同复用原则：放在同一个包、模块或库里的东西是否会被同一批使用者一起复用？无关 helper 堆到 shared/common 会制造无谓依赖。
- LSP，里氏替换原则：实现类、策略、适配器或 mock 是否能替换接口而不破坏调用方预期？
- CCP，共同闭包原则：因同一个业务变化而一起改的代码是否被放在一起？频繁跨模块同步修改说明边界可能切错。

常见架构风格判据：

- 分层架构：上层是否只依赖下层，是否出现 handler 直接访问 storage、repository 反调 service 等绕行。
- 六边形/端口适配器：领域核心是否通过 port 依赖外部能力，外部 SDK 和框架是否被隔离在 adapter。
- Clean Architecture：业务规则是否独立于 UI、数据库、框架和外部服务。
- DDD 分层：实体、值对象、领域服务、应用服务、仓储和防腐层是否各自承担清晰语义。
- 事件驱动：事件是否表达已经发生的业务事实，发布和持久化是否一致，消费者是否幂等。
- CQRS：读写模型分离是否由性能或一致性需求驱动，还是制造了双写和同步成本。
- Pipeline：阶段顺序、输入输出和失败处理是否清晰，是否有阶段偷偷修改全局状态。

常见设计模式判据：

- Strategy：多策略选择是否消除了核心分支膨胀，还是只把简单 if 分散到多个文件。
- Adapter：是否隔离外部系统差异，还是把外部 SDK 类型泄露到业务层。
- Facade：是否简化跨子系统调用，还是变成新的全能 service。
- Factory/Builder：是否封装复杂构造和不变量，还是隐藏了必要依赖。
- Template Method/Hook：扩展点是否稳定，还是让子类或回调影响主流程不变量。
- Observer/Event：是否解耦副作用，还是让关键一致性路径变得不可追踪。
- Command：是否清晰表达用户意图和可重试边界，还是把贫血参数包伪装成对象。

## 代码风格和可维护性

- 函数长度是否明显超过附近承担类似职责的函数？
- 函数是否混合了校验、业务规则、持久化、远程调用、日志和展示？
- 抽 helper 是否表达了稳定概念，还是只制造了没有领域意义的小跳转？
- 参数数量是否超过项目和语言习惯？
- 是否存在多处一起传递的一组参数？
- boolean 参数是否在暗中控制模式，应该改为命名选项或拆分函数？
- 导出函数、类、结构体、字段、常量、包或模块是否真的需要跨边界访问？
- public API 是否暴露了内部状态、存储细节、重试策略、缓存 key、框架类型或传输细节？
- 错误是否携带足够上下文，同时保留调用方依赖的类型或哨兵判断？
- 命名是否符合项目已有词汇？
- 是否存在跨文件、测试、迁移或适配器的重复逻辑？
- 代码是在直接解决实际问题，还是在没有第二个真实用例时构建通用抽象？
- 实现是否过于简单，遗漏校验、状态处理或失败处理？
- 实现是否过于复杂，引入并发、缓存、反射、代码生成、全局状态或框架 hook 却没有明确收益？

## 注释和日志

- 每条注释在变更后是否仍与代码一致？
- 注释是否解释了约束存在的原因，而不是复述代码在做什么？
- TODO/FIXME 是否可执行、有人负责，或关联真实 issue？
- 边界穿越、外部调用、状态迁移、重试、永久失败和异常兜底路径是否有日志？
- 日志是否包含诊断所需标识，同时避免泄露密钥、token、个人信息或敏感业务数据？
- 错误是否被吞掉，只打日志不返回，或返回时缺少上下文？
- 是否存在热循环或高频路径日志过多的问题？

## 单测质量

- 如果回滚这个 patch，哪一个行为会坏？这个行为是否被测试覆盖？
- 是否覆盖失败模式、边界值、空输入、权限错误、部分失败、超时、并发和幂等场景？
- 测试断言的是外部可见行为，还是私有实现细节？
- mock 是否过度宽松，允许任意调用顺序或任意参数？
- snapshot 或 golden 文件是否有实质断言价值？
- 生产代码是否为了测试方便变得更弱、更可配置或更不严格？
- 测试更新是因为行为确实改变，还是因为旧断言不方便？
- 如果本次变更声称修复 bug，测试是否会在该 bug 存在时失败？
- 是否实际运行了相关测试、静态检查或构建？如果没有，原因是什么，残余风险是什么？

## 功能正确性

- 实现是否满足用户可见需求和已有契约？
- 输入校验是否完整，并放在正确边界？
- nil、null、空、零值、负数、重复值、未知值和畸形值是否按项目约定处理？
- 权限检查、所有权检查、租户边界和 feature flag 是否保留？
- 错误是否以正确状态、正确重试语义传给调用方？
- cancel、context deadline、信号处理和清理路径是否被尊重？
- migration、默认值、配置兼容和灰度路径是否处理？
- public API、CLI、配置文件、持久化数据和 wire format 是否保持向后兼容？

## 性能

- 是否在循环里新增重复数据库调用、远程调用、文件读取或全量扫描？
- 大输入是否有边界、流式处理、分页或分块？
- 缓存是否正确失效，缓存 key 是否稳定？
- 是否持锁执行 I/O、日志、序列化或回调？
- 并发是否提升吞吐，同时避免竞态或无界 goroutine/task？
- 昂贵计算是否被安全复用，避免重复执行？

## 一致性和状态

- 相关状态更新是否原子？
- 事务边界是否清晰并符合已有模式？
- 两个请求并发时是否会产生重复、丢失或陈旧状态？
- 重试、webhook、job 或分布式调用是否需要幂等？
- 最终一致性假设是否明确，并且调用方能承受？
- 缓存、索引、反规范化字段和派生状态是否一起更新？
- outbox、队列或事件发送是否与持久化成功失败保持一致？

## 问题输出质量

一条合格问题应包含：

- 严重级别：Critical、High、Medium 或 Low。
- 位置：尽量给出精确文件和行号。
- 证据：代码当前实际做了什么。
- 影响：会如何失败，或如何增加长期成本。
- 修复方向：足够具体，可执行，但不必替用户写完整补丁。

避免只输出个人偏好。除非用户明确要求风格打磨，否则偏好类意见只作为备注。

完整审查回复至少包含三条有价值反馈：

- 明确问题优先，按严重级别排序。
- 如果确认问题不足三条，不要捏造缺陷；用架构贴合判断、测试缺口、残余风险、可维护性建议、验证结果或正向确认补足。
- 每条反馈都必须基于代码或审查动作，避免空泛建议。

## 输出裁剪

- 有明确问题时，先输出问题，不要先写总结。
- 架构形态只在架构边界、模块职责、依赖方向或新增抽象相关时输出。
- 测试部分只在运行了测试、无法运行测试、或发现测试缺口时输出。
- 开放问题为空时省略。
- 没有发现问题时，直接说明未发现明确问题，再说明未验证内容和残余风险。
- 即使没有明确问题，也要通过“其他有价值反馈”或等价结构补足至少三条反馈。
